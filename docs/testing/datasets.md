# Источники тестовых данных

Суммарно в тестовом контуре использовались `348` интеграционных запусков и отдельный набор unit-проверок по изолированным модулям.

## Источники данных
1. Внутренний набор безопасных URL.
   Использовались корпоративные, учебные и нейтральные сайты без признаков фишинга.
2. Внутренний набор фишинговых URL из выборки.
   Использовались адреса с типичными для фишинга признаками: подмена бренда, сбор credentials, редиректы, бесплатный хостинг, подозрительные формы.
3. Синтетические URL для граничных и спорных случаев.
   Включались поддомены, сокращатели ссылок, многошаговые редиректы, смешанные алфавиты, похожие домены и URL с редкими структурами.
4. Ручные сценарии действий пользователя.
   Проверялись popup, blocked page, кнопки предупреждения, переходы по ссылкам, временное разрешение домена и работа репорта.

## Где эти данные лежат в проекте
- `tests/integration/data/environments.json`
- `tests/integration/data/safe-urls.json`
- `tests/integration/data/phishing-urls.json`
- `tests/integration/data/synthetic-cases.json`
- `tests/integration/data/manual-ui-scenarios.json`

## Как данные используются
- `safe-urls.json` разворачивается в `144` интеграционных запуска.
- `phishing-urls.json` разворачивается в `55` интеграционных запуска.
- `synthetic-cases.json` разворачивается в `137` интеграционных запуска.
- `manual-ui-scenarios.json` разворачивается в `12` интеграционных запуска.

Суммарно это даёт `348` сценариев, которые исполняются раннером `tests/integration/run-integration.mjs`.

## Причина выбора таких источников
1. Безопасные URL дают оценку ложных срабатываний.
2. Фишинговые URL дают оценку полноты обнаружения угроз.
3. Синтетические кейсы закрывают редкие, но критичные углы логики.
4. Ручные сценарии позволяют оценить риск ошибки человека и устойчивость UI.

## Контрольные группы
- Безопасные URL: `144`
- Фишинговые URL: `55`
- Сценарии с отказом ML и fallback: `24`
- Остальные интеграционные прогоны: граничные и пользовательские сценарии, включая редиректы и спорные случаи
